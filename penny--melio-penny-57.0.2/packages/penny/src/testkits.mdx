## The Problem

Writing tests using Penny components can be tricky because internal DOM structure and implementation details can change, even when the component’s behavior stays the same.
If your tests rely on those details, they can break frequently, creating extra maintenance work and making test results less reliable.

## The Solution
A Penny Testkit is created by a function that returns an object that wraps a component and **provides methods for interacting with it and checking its state**.
By handling the component's internal details, testkits let you focus on **what the component does rather than how it works.**
This gives you a **unified, high-level interface** for writing tests with Penny components and helps your tests stay **robust, readable, and maintainable**, even as the system evolves.

**Before**
<div style={{ marginTop: '-20px' }} />
```ts
import { screen, render } from '@testing-library/react';

it('should render form text field in view mode', () => {
  render(
    <FormField
      data-testid="custom-id"
      render={(props) => <TextField {...props} />}
    />
  );
  const textFormFieldLabelElement = screen.getByTestId('custom-id-label');

  expect(textFormFieldLabelElement.getAttribute('data-view-mode')).toBe('true')
});
```
<div style={{ marginTop: '-20px' }} />

In this approach, you have to understand internal implementation details of the `FormField` component.
For example, to check if the field is in view mode, you need to know:

- That the label’s `data-testid` is generated by appending `-label` to the main input’s test ID.
- That the “view mode” state is indicated via a `data-view-mode` attribute on the label.

<div style={{ marginBottom: '40px' }} />

**After**
<div style={{ marginTop: '-20px' }} />
```ts
import { render } from '@testing-library/react';
import { createFormFieldTestKit } from '@melio/penny-testkit-rtl';

it('should render form text field in view mode', () => {
  const textFormFieldTestKit = createFormFieldTestKit('custom-id', { inputType: 'TextField' });
  render(
    <FormField
      data-testid="custom-id"
      render={(props) => <TextField {...props} />}
    />
  );

  expect(textFormFieldTestKit.getIsViewMode()).toBeTruthy();
});
```
<div style={{ marginTop: '-20px' }} />

With a testkit, you can directly call `getIsViewMode()`.
You don’t need to know how the component is implemented, which DOM node represents the label, or which attributes indicate the state.
The testkit encapsulates these details, letting you focus on testing behavior.

## Scope and Coverage
- **Focused on common scenarios:** Testkits provide methods for typical use cases and common states of a component.
- **Not exhaustive:** They do not cover every edge case or internal element.
- **Internal elements are not exposed:** Most internal elements of a component are not directly accessible. You should not check for the existence of elements in the DOM.
instead, use the testkit’s methods to verify their state or content.
If for some reason you need to test something on a specific internal element, you can use the component’s data-testid as an escape hatch.

    ✅ **Do**
    <div style={{ marginTop: '-10px' }} />
    ```ts
    expect(formFieldTestKit.getIsInvalid()).toBeTruthy();
    expect(formFieldTestKit.getErrorMessage()).toBe('some error message');
    ```

    ❌ **Don't**
    <div style={{ marginTop: '-10px' }} />
    ```ts
    const formFieldErrorMessageElement = screen.getByTestId('form-field-error-message')
    expect(formFieldErrorMessageElement).toBeInDocument();
    ```

### Supported components
Not all components currently provide a testkit. For components that do, you can find a dedicated **RTL** or **Playwright** Testkit file under the component’s section in Storybook, including the full API and usage examples.
