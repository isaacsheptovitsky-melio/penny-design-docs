## Core principle

**Buttons perform actions. Links navigate.**

Every interactive element in the UI should map to exactly one of these roles:

- **Button** (`<button>`) — the user is doing something that changes state: submitting a payment, confirming a deletion, saving a draft, activating a feature. The result is a mutation.
- **Link** (`<a>`) — the user is going somewhere: a different page, a drawer, an external resource, a modal that merely shows information. The result is navigation.

This distinction is not visual — it is semantic. A link styled to look like a button is still a link. A button styled as underlined text is still a button. The element type drives screen reader announcements, keyboard behavior, and browser affordances.

## Implementation & accessibility

Even when a link is styled to visually resemble a button (or vice versa), the underlying HTML must match the element's function. Screen readers rely on specific tags to tell users whether they are about to perform an action or navigate to a new location.

### As a button

The `<button>` tag is used for actionable elements. This informs screen readers that interacting with the element will trigger an in-page event, such as submitting data or opening a modal. Penny's `Button` renders a native `<button>` by default.

<Canvas of={BvLStories.AsButtonExample} sourceState="shown" />

### As a link

The `<a>` (anchor) tag is used for navigation. This informs screen readers that the element is a link, allowing users to anticipate a change in URL or page context. Pass a `link` prop and `Button` renders an `<a>` while keeping the button styling.

## Do's and don'ts

### Correct usage

<Canvas of={BvLStories.DoExamples} />

<Do>use a button for a data-altering action — saving information and moving forward, submitting a form, or saving a new vendor. The click changes application state.</Do>

<Do>use button styling for a navigation link only when it is the singular, primary focus of its area or page (for example, the one call-to-action in an empty state).</Do>

### Incorrect usage

<Canvas of={BvLStories.DontExamples} />

<Dont>use a button for a link that simply loads a new page without saving or changing data ("Read more", "View roles") — it confuses the user's expectation of "submitting" something.</Dont>

<Dont>style a link as a button when it competes with actual functional buttons (like "Pay" or "Save"), or when it clutters the visual hierarchy of a dense interface.</Dont>
