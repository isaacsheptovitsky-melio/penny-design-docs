## Overview

The feedback pattern is built on a single component — **Section Banner's Compact variant** — placed at the location most relevant to the error. The flow below shows when to use each placement.

<Canvas of={FeedbackStories.FeedbackOverview} />

<Canvas of={FeedbackStories.SectionBannerCompact} />

## Global feedback

Use global placement when an issue affects the **entire screen or process**. Position the Section Banner at the top of the screen, interrupting the user's flow to demand immediate attention.

<Canvas of={FeedbackStories.GlobalFeedbackPlacement} />

<Do>place the banner at the very top of the page or drawer, above all other content, so it is immediately visible on arrival.</Do>

<Dont>bury a page-level error inside a form section — users who scroll past it may never see it.</Dont>

## Local feedback

Use local placement when an action is **isolated to a specific section** and keeping feedback near the trigger provides necessary context.

<Callout variant="note">

**Global placement is the default.** Use local placement only when the affected section is clearly distinguishable and the error cannot be attributed to the whole page.

</Callout>

Position the Section Banner directly above the section or element where the issue occurs.

<Canvas of={FeedbackStories.LocalFeedbackPlacement} />

<Do>place the banner directly above the section heading where the issue originates.</Do>

<Dont>show more than one banner per section — especially multiple critical banners at once. Consolidate into a single list.</Dont>

## Form validation

Form validation runs in two sequential stages: first checking input format in the browser, then checking data quality on the server. Each stage has a distinct visual treatment.

<Canvas of={FeedbackStories.FormValidationOverview} />

### Client-side validation

Runs **locally in the browser** before data is sent to the server.

- **When to use:** Required fields left empty, or formatting errors (incomplete routing number, missing vendor name).
- **Behavior:** The request is not sent to the server — the input is technically incorrect.
- **Visual style:** Use the **critical state** of the specific Form Field component, not a banner.

<Canvas of={FeedbackStories.ClientSideValidation} />

<Do>flag each invalid field individually using the Form Field critical state, with a specific inline error message per field.</Do>

<Dont>show a section banner for client-side errors — reserve banners for server responses.</Dont>

### Server-side validation

Runs **after the form is successfully submitted** and the server processes the data.

- **When to use:** Data format is correct but content fails (vendor already exists, payment exceeds limit).
- **Behavior:** The server returns a validation response indicating issues.
- **Visual style:** Use a **Section Banner** placed directly above the form heading.

<Canvas of={FeedbackStories.ServerSideValidation} />

### Validation logic & placement

Follow this order of operations on every form submission:

1. **Validate client-side first** — check for formatting issues. Highlight invalid fields with the Form Field critical state.
2. **Validate server-side second** — once the form reaches the server, display any returned errors using a Section Banner.
3. **Banner placement rules:**
   - Place the banner directly **above the form heading** (not above the submit button)
   - If there are multiple issues, **consolidate them into a single banner** with a list — never stack multiple banners

## Actions and dismissal

- **Always include an action or explanation** when the user must resolve the error — the banner should tell them what to do next.
- **Do not allow dismissal** for critical errors or when an action is required. Dismissal is appropriate only when the message is informational and no action is needed.

## Implementation checklist

When adding feedback to a flow, verify each of these:

- [ ] Is the error **page-level** (affects the whole screen)? → use Global placement — Section Banner at the top of the page.
- [ ] Is the error **section-level** (affects one isolated area)? → use Local placement — Section Banner above that section's heading.
- [ ] Is the error a **client-side format issue** (empty required field, bad format)? → use the Form Field's `status="critical"` inline, not a banner.
- [ ] Is the error a **server-side data issue** (vendor exists, limit exceeded)? → use Section Banner above the form heading.
- [ ] Are there multiple server-side issues? → consolidate into **one** Section Banner with a bullet list. Never stack multiple banners.
- [ ] Does the error require user action? → include a clear CTA in the banner. Do not allow dismissal.
- [ ] Is the error informational with no required action? → dismissal may be allowed.
- [ ] After the user resolves the error and resubmits successfully → remove the banner and show a success Toast.

## Components used

- **`SectionBanner` (`variant="critical"`, `size="compact"`)** — Global and local error banners
- **Form Field `status="critical"`** — Client-side field-level validation errors
- **`Toast` (`variant="success"`)** — Post-action success confirmation

## Error handling in API calls

When a form submission fails:

1. **Set loading state** on the submit button (`isLoading={true}`).
2. **Call the API.**
3. **On success:** clear the form, close the modal/flow, show a success Toast.
4. **On failure:** set `isLoading={false}`, display a Section Banner above the form heading with the error message. Do not close the modal.

```tsx
const [isSubmitting, setIsSubmitting] = useState(false);
const [errorMessage, setErrorMessage] = useState<string | null>(null);

const handleSubmit = async () => {
  setIsSubmitting(true);
  setErrorMessage(null);
  try {
    await submitPayment(formData);
    onClose();
    showSuccessToast('Payment scheduled');
  } catch (err) {
    setIsSubmitting(false);
    setErrorMessage('Payment failed. Please check your account balance and try again.');
  }
};

// In the render:
{errorMessage && (
  <SectionBanner
    variant="critical"
    size="compact"
    description={errorMessage}
  />
)}
```

## Accessibility

### Error announcement

- **Role:** Section Banner uses `role="alert"` internally, which causes screen readers to announce the message immediately when it is injected into the DOM — no focus management required for the banner itself.
- **Timing:** Inject the banner into the DOM **after** the API response resolves, not before. Pre-rendering a hidden banner and toggling visibility prevents the `alert` role from firing.

### Form field errors (client-side)

- Each Form Field in error state must have `aria-invalid="true"` and `aria-describedby` pointing to the error message element.
- The error message must be programmatically associated — not just visually adjacent.
- Do not rely on color alone to indicate an error. The field border, icon, and text message must all change.

### Focus management

- After client-side validation fires, move focus to the **first invalid field** so keyboard users know where to begin correcting.
- After server-side errors appear, move focus to the **Section Banner** so the error is announced immediately.
- After successful submission, move focus to the **main content region** or the newly loaded page heading.

### Color contrast

- Critical Section Banner: text and icon colors meet WCAG 2.1 AA (4.5:1 for body text, 3:1 for large text / icons) against the `semantic.fill.critical.secondary` background.
- Do not use color alone to distinguish error from success states — always pair color with an icon and text label.
