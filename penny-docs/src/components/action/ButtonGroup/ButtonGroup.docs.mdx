## Usage

- Use a Button Group to **bind closely-related actions** into one visual unit — `variant`, `size`, `isLoading`, and `isDisabled` are set once on the container and inherited by every child.
- The most common use is the **Split Button**: a default action paired with a menu of related options.
- Keep the default action **predictable** — users shouldn't have to open the menu to understand what the component does.
- Constrain group usage to the regular, positive variants — **`primary`** and **`tertiary`**. Reserve `critical` for genuinely destructive groupings.

<Callout variant="tip">Make the default action self-explanatory. If the primary button's purpose only becomes clear after opening the dropdown, the grouping is wrong.</Callout>

## Variants

The `variant` prop is applied uniformly to **every** button in the group — you set it once on the container, not on each child. Keep group usage to the regular, positive variants.

### Icon buttons

A group can hold `IconButton`s instead of text buttons. The seams between cells are merged the same way.

### Shared state

`isLoading` and `isDisabled` cascade to the whole group — every button reflects the container's state at once.

## Sizes

Three sizes are available — `medium` is the default. The `size` prop sizes the entire group together.

## Split Button

The Split Button is the primary pattern built on Button Group: it **provides a default action while grouping a set of secondary, related actions into a dropdown menu**. Compose one by placing a `Button`, an `IconButton`, and a `Menu` together under a single `ButtonGroup`.

<Canvas of={ButtonGroupStories.AsSplitButton} />

Some good split-button candidates:

- **Save** → *Save as template*
- **Create bill** → *Import from spreadsheet*, *Add from Gmail*

### When to use each variant

The Split Button is available in every Button variant, but keep usage to the regular, positive ones.

<Story of={ButtonGroupStories.SplitButtonGuide} />

<Callout variant="warning">`critical` and `critical-secondary` split buttons are possible but rarely appropriate — hiding destructive options behind a dropdown makes them easy to trigger by accident. You'd better have a good reason.</Callout>

## Mobile usage

On mobile, a Split Button's default and secondary options surface in a **bottom sheet** rather than an anchored dropdown menu — the tap targets are larger and easier to reach with the thumb.
