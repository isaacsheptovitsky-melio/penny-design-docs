## Callouts

Four variants cover every tone of documentation annotation. Use the right one to set the correct reader expectation.

<Callout variant="note">**Note / Tip** — A neutral informational annotation or usage suggestion. Friendly and constructive. `variant="note"` and `variant="tip"` are aliases — both work identically.</Callout>

<Callout variant="success">**Success** — Reinforces correct usage or a recommended pattern. Use `variant="success"`.</Callout>

<Callout variant="warning">**Warning / Caution** — Something to watch out for. Non-blocking but important. `variant="warning"` and `variant="caution"` are aliases — both work identically.</Callout>

<Callout variant="critical">**Critical** — Signals danger or an irreversible consequence. Use sparingly. Use `variant="critical"`.</Callout>

Icons are inline SVGs in each variant's semantic color. **Never use emojis inside Callouts** — the component renders Penny-style SVG icons automatically.

## Logos

All logo assets live in `src/assets/`. Use the variant that matches the background — never modify, recolor, or stretch any logo.

<Canvas of={DesignGuidelinesStories.LogosShowcase} />

### Penny

Three color variants × two layouts. Use **Black** by default on light backgrounds.

- **Black** (`penny-symbol-black` / `penny-logo-black-*`) — White and light backgrounds (default)
- **Lylac** (`penny-symbol-lylac` / `penny-logo-lylac-*`) — Light grey surfaces where brand color adds warmth
- **White** (`penny-symbol-white` / `penny-logo-white-*`) — Dark (`#18191b`) or brand purple (`#7849FF`) backgrounds

Two layout options:
- **Symbol only** (`40px`) — use in tight spaces: favicons, shortcut tiles, avatar slots.
- **Logo + tagline** — use in headers, marketing sections, and onboarding. Available in **horizontal** and **vertical** stacks.

### Melio

Two logo types are available:

- **Full Logo** (wordmark) — variants: Black · Purple · White — Marketing, headers, onboarding; use whenever the Melio name should be visible
- **App Symbol** (rounded square) — variants: Black · Purple · White — Product icons, app launchers, tight spaces

Color rules:
- **Black** — on white or light backgrounds
- **Purple** — on light grey surfaces for a branded feel
- **White** — on dark (`#18191b`) or brand-colored backgrounds

### Usage rules

- **Never** recolor, rotate, scale non-uniformly, or add effects to any logo.
- **Never** place the Penny or Melio logo on a background with insufficient contrast. Test visually — the symbol must be clearly legible.
- **Minimum size** — the Penny symbol must not be rendered below 24px. Below that size, substitute with a text label.
- **Clear space** — maintain at least 1× the symbol height as whitespace around any logo in composed layouts.
- **Shortcut tiles** — the black Penny symbol is used automatically for all internal (`/?path=…`) links. External links show the site favicon.

## Inline Overlay Annotations

A positioning pattern for annotating specific points in a rendered UI mockup — prop values, placement conventions, visual states — without distorting the layout. The visual is the back layer; the annotation is a transparent front layer.

**When to use** — Labelling a specific element inside a live Canvas story (a button variant, a layout rule, a token reference). Use when you need the dot to point *at* something in context, and `<PartCallout>` would change the visual layout.

**When not to use** — Wrapping a standalone element you own entirely (use `<PartCallout>` instead). Never annotate something you can explain just as well with a prose caption or `<Callout>`.

<Canvas of={DesignGuidelinesStories.OverlayAnnotationExample} />

### Core pattern

Wrap the card and the annotation together in a `position: relative` container. The card uses `overflow: hidden` to clip its own children — but the annotation is a **sibling**, not a child, so it is never clipped by the card. Set `pointer-events: none` so the annotation never blocks interaction.

```tsx
<div style={{ position: 'relative' }}>

  {/* Visual card — overflow:hidden clips only its own children */}
  <div style={{ background: '#fff', border: '1px solid #E2E8F0', overflow: 'hidden' }}>
    {/* your UI mockup */}
  </div>

  {/* Annotation sibling — never clipped, always on top */}
  <div style={{
    position: 'absolute',
    top: `${buttonCenterY}px`,
    /* left or right — see patterns below */
    transform: 'translateY(-50%)',
    display: 'flex',
    alignItems: 'center',
    gap: '4px',
    pointerEvents: 'none',
  }}>
    {/* dot → line → label  (or  label → line → dot) */}
  </div>

</div>
```

### Dot placement rule

**Always place the dot in whitespace — never over text or interactive content.**

The dot marks where the annotation *points*, not where the label lives. Safe whitespace locations:

- **Right padding strip** — The gap between a right-aligned button and the card's right border
- **Left padding strip** — The gap between the card's left border and a left-aligned button
- **Space-between gap** — Empty area between two `justifyContent: space-between` siblings

### Right-side annotation

Use when the annotated element is near the **right edge** of the card. DOM order `[dot][line][label]` — elements flow left-to-right so the label exits the card on the right.

`left: 'calc(100% - Xpx)'` anchors the dot's left edge inside the right padding strip. Set X to half the card's right padding (e.g. `10px` for a `20px` right pad).

```tsx
{/* Dot in right padding, line + label extend outward to the right */}
<div style={{
  position: 'absolute',
  top: `${elementCenterY}px`,
  left: 'calc(100% - 10px)',   // 10px = half of 20px right padding
  transform: 'translateY(-50%)',
  display: 'flex',
  alignItems: 'center',
  gap: '4px',
  pointerEvents: 'none',
}}>
  <div style={{ width: '3px', height: '3px', borderRadius: '50%', background: '#9CA3AF', border: '1px solid white', flexShrink: 0 }} />
  <div style={{ width: '32px', height: '1px', background: '#D1D5DB', flexShrink: 0 }} />
  <span style={{ fontFamily: '"SFMono-Regular", Consolas, monospace', fontSize: '10px', color: '#6B7280', whiteSpace: 'nowrap' }}>
    label text
  </span>
</div>
```

### Left-side annotation

Use when the annotated element is near the **left edge** of the card. DOM order `[label][line][dot]` — dot is the rightmost element, so it lands in the left padding strip while the label exits the card on the left.

`right: 'calc(100% - Xpx)'` anchors the container's right edge inside the left padding strip. Set X to half the card's left padding (e.g. `14px` for a `24px` left pad).

```tsx
{/* Label extends outward to the left, line + dot land in left padding */}
<div style={{
  position: 'absolute',
  top: `${elementCenterY}px`,
  right: 'calc(100% - 14px)',  // 14px = ~half of 24px left padding
  transform: 'translateY(-50%)',
  display: 'flex',
  alignItems: 'center',
  gap: '4px',
  pointerEvents: 'none',
}}>
  <span style={{ fontFamily: '"SFMono-Regular", Consolas, monospace', fontSize: '10px', color: '#6B7280', whiteSpace: 'nowrap' }}>
    label text
  </span>
  <div style={{ width: '32px', height: '1px', background: '#D1D5DB', flexShrink: 0 }} />
  <div style={{ width: '3px', height: '3px', borderRadius: '50%', background: '#9CA3AF', border: '1px solid white', flexShrink: 0 }} />
</div>
```

### Calculating `top`

`top` should land at the vertical centre of the element being annotated, measured from the relative wrapper's top edge (= the card's top border).

```
top = (sum of heights of all elements above the target row)
    + target row top-padding
    + (target element height / 2)
```

### Visual spec

- **Canvas bg** — `#EDF2F7`. One step darker than the card visual panels (`#F8F9FA`) so cards lift off the tray. Set per-story in `preview-head.html` using a `:has()` scoped rule.

## DocTabs

The three-tab wrapper used on every component page. Wraps all content below the `<PageHero>`. Always use exactly three tabs in this order: **Design**, **Development**, **ADA**.

```tsx
import { DocTabs, DocTab } from '@/storybook-utils/DocTabs';

<DocTabs links={<ShortcutTiles>…</ShortcutTiles>}>
  <DocTab label="Design">
    {/* Canvas, Controls, Usage, Variants, Sizes, patterns */}
  </DocTab>
  <DocTab label="Development">
    {/* Anatomy (if needed), Component as link, Props */}
  </DocTab>
  <DocTab label="ADA">
    {/* Accessibility */}
  </DocTab>
</DocTabs>
```

- The `links` prop renders a `<ShortcutTiles>` bar pinned above the tab strip.
- Non-props Controls categories (accessibility, events, tests) auto-collapse via the `preview-head.html` MutationObserver — no per-story config needed.
- All markdown content (headings, bullets, `<Canvas>`, `<Callout>`, etc.) works inside `<DocTab>` as long as there are blank lines between the JSX tag and the markdown.
- Do **not** use `---` horizontal rules inside `<DocTab>` — section spacing is sufficient.

### Tab content order (Design tab)

The canonical section order inside the Design tab:

1. `<Canvas of={Stories.Playground} />` — interactive hero story
2. `<Controls of={Stories.Playground} />` — props table directly below, **no heading**
3. `## Usage` — bullet list of do/don't rules
4. `## Variants` — variant table + hierarchy cards + states grid
5. `## Sizes` — if the component has sizes
6. `## [Feature]` — one section per significant feature
7. `## Desktop usage` / `## Mobile usage` — layout rules if applicable

## Inline Rules — `<Do>` and `<Dont>`

Used for concise do/don't rules inside prose — distinct from `<DoDont>` image cards. These render an inline SVG check (green `#028838`) or cross (red `#D80E25`) icon before the rule text.

```tsx
import { Do, Dont } from '@/storybook-utils/RuleItem';

<Do>place the delete action at the row level so it's always scoped to a single item.</Do>

<Dont>place a bulk "Delete all" action unless the user has first selected specific items.</Dont>
```

**Rules:**
- Group related do/don'ts under a `###` subheading. At most 3 per subheading.
- **Never use emoji** (✅ / ❌) — always use `<Do>` and `<Dont>` so the visual stays consistent with the Penny icon system.
- `<Do>` / `<Dont>` = prose rules in the **Usage** or **Trigger placement** sections.
- `<DoDont>` = image cards showing product UI mockups in **Desktop/Mobile usage** sections.

## Font rules

Two completely separate font contexts govern this Storybook:

### Docs UI (MDX prose, storybook-utils components)

Inherits **Nunito Sans** from `.sbdocs { font-family: 'Nunito Sans', sans-serif }` in `preview-head.html`. This uses CSS inheritance — **no `!important`, no `*` selector** — so Penny component themes (Poppins) win naturally.

**Rule: Never set `fontFamily` in any inline style** on docs-UI elements (MDX headings, body text, `<Callout>`, `<VariantCard>` description text, table cells, `<SectionBox>` headers). They inherit Nunito Sans automatically.

### Story render() mockups

The outer `<div>` wrapper of every story's `render()` function **must** set `fontFamily: 'Poppins, sans-serif'` so all mockup text (simulated UI labels, card text, drawer headers) inherits Poppins instead of the docs Nunito Sans.

```tsx
// Correct — Poppins flows into all mockup children
render: () => (
  <div style={{ padding: '32px', fontFamily: 'Poppins, sans-serif' }}>
    <p style={{ ... }}>This text inherits Poppins ✓</p>
    <Button label="Penny component — already Poppins from theme" />
  </div>
)

// Wrong — mockup text picks up Nunito Sans from .sbdocs
render: () => (
  <div style={{ padding: '32px' }}>
    <p style={{ ... }}>This text is Nunito Sans ✗</p>
  </div>
)
```

Penny components inside the render already receive Poppins through their Chakra theme — the `fontFamily` on the outer div is only for your surrounding mockup HTML.

## Canvas usage

**Playground story** — `<Canvas of={Stories.Playground} />` followed immediately by `<Controls of={Stories.Playground} />` with no heading between them. The `Playground` story's `parameters.controls.include` lists only prop-category argTypes so the Controls table stays focused.

**Layout stories** (Hierarchy, States, Inverse) — set `parameters: { controls: { disable: true }, docs: { canvas: { sourceState: 'none' } } }` in the story. Reference with `<Canvas of={…} />` or `<Story of={…} />` from MDX.

**Feature demos** (Sizes, Elements, FullWidth) — use default `sourceState` (shows "Show code" button). Reference with `<Canvas of={…} />`.

```tsx
// Layout story — no controls, no source
export const VariantHierarchy: Story = {
  parameters: {
    controls: { disable: true },
    docs: { canvas: { sourceState: 'none' } },
  },
  render: () => ( /* … */ ),
};

// Playground — controls scoped to props only
export const Playground: Story = {
  parameters: {
    controls: {
      include: ['label', 'variant', 'size', 'isDisabled'],
    },
  },
};
```
